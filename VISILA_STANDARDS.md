# Visila Standards Engine
### The complete source of truth for every app built on Visila

Every app generated by Visila inherits this system by default. This is not a checklist. It is a worldview — baked into the generation layer so that every non-developer who builds on Visila ships a world-class product from day one.

## Part 1 — Design
*Jony Ive standard*

- Remove everything that isn't essential. If an element doesn't help the user accomplish their goal, it doesn't exist.
- The best interface is the one that disappears. The user focuses on their task, not the UI.
- Every screen has one primary purpose. One dominant action. One clear next step.
- Consistency is not repetition — it is trust. Same patterns, same spacing, same behavior throughout.
- Details are not details. They make the product. Spacing, alignment, type weight, border radius — all intentional.
- Never ship unfinished UI. Empty states, loading states, error states — all designed with the same care as the happy path.
- Motion has meaning. Animate only to orient, confirm, or delight — never to show off.
- Products should feel inevitable — like they couldn't have been designed any other way.

**Visila Unified Design System — Color Tokens**
Single source of truth for all colors on visila.com.
Defined in src/styles/tokens.css. Three layers:

**Layer 1 — Primitives (never use in components)**
Raw values. `--primitive-flame`, `--primitive-ink`, `--primitive-paper`,
`--primitive-gray-1` through `-6`, `--primitive-dark-1` through `-6`,
status primitives. Reference only from semantic tokens.

**Layer 2 — Semantic tokens (always use in components)**

Brand:
  `--color-brand`           Flame #FF1F6E — CTAs, active states, focus
  `--color-brand-hover`     Mid #CC0055 — hover on brand elements
  `--color-brand-subtle`    Highlight #FF80AA — tints, backgrounds
  `--color-brand-dark`      Dark #8B0040 — pressed states

Backgrounds (Apple HIG two-layer depth):
  `--color-bg-base`         Page backgrounds
  `--color-bg-raised`       Cards, panels, inputs
  `--color-bg-overlay`      Modals, popovers, drawers
  `--color-bg-sunken`       Wells, code blocks, recessed areas

Text (WCAG AA — 4.5:1 min, 7:1 target for body):
  `--color-text-primary`    Body text, headings
  `--color-text-secondary`  Metadata, captions, labels
  `--color-text-tertiary`   Hints, disabled text
  `--color-text-placeholder` Input placeholders
  `--color-text-inverse`    Text on dark/brand backgrounds
  `--color-text-brand`      Links, accents, interactive text

Borders:
  `--color-border`          Default input and card borders
  `--color-border-strong`   Emphasized borders, focus rings context
  `--color-separator`       Divider lines between sections

Status:
  `--color-success` / `--color-warning` / `--color-error` / `--color-info`
  Never override these — must stay consistent.

Focus:
  `--color-focus-ring`      Always `--color-brand` (Flame)

**Layer 3 — Dark mode**
Defined under `.dark` selector. Semantic tokens only — primitives
never change. Colors are NOT inversions of light mode.
Source: Apple HIG Dark Mode documentation.

**Rules:**
- Always reference semantic tokens — never primitives, never hex
- `--color-brand` is always Flame on visila.com — never override
- Status tokens are never overridden anywhere
- Generated apps use their own token set (`api/_designSystem.ts`)
  which follows the same naming convention
- Verify contrast at https://webaim.org/resources/contrastchecker/

**Design tokens applied to generated apps:**
- Fonts: Playfair Display (headings) + DM Mono (body/UI)
- Colors: per-app palette generated by `api/_designSystem.ts` (9 categories)
- WCAG contrast: primaryColor darkened 30% for any text or border use on light backgrounds
- Button text: white or black — whichever passes 4.5:1 against the button color

## Part 2 — Copy & Voice
*Apple content strategist standard*

- **8th grade reading level** on every screen. Short sentences. Active voice. No jargon.
- Headlines state the outcome, not the feature. "Your app, live in minutes" not "Automated deployment pipeline."
- CTAs are specific verbs. "Start building" not "Get started." "Send invite" not "Submit."
- Error messages are human. They say what went wrong and exactly how to fix it. Never raw technical errors.
- Empty states guide — headline, one sentence of context, one clear action.
- Microcopy earns trust. Placeholder text teaches. Confirmation messages reassure.
- Every word is load-bearing. If removing a word doesn't change the meaning, remove it.
- One consistent voice: confident, warm, direct. Never corporate. Never casually unprofessional.

## Part 3 — Marketing & Positioning
*Seth Godin standard*

- Every app needs a purple cow — one thing that makes it remarkable and worth talking about.
- Positioning must be specific. "For independent tattoo artists" beats "for everyone."
- The landing page headline must pass the **5-second test**: a stranger understands what it does and who it's for within 5 seconds.
- Features are not benefits. Lead with the transformation: "From idea to live app in minutes."
- The story is the product. Why does this exist? Who is it for? What changes when they use it?
- Build in natural share moments — celebrate milestones, make success visible, give users something to show off.

## Part 4 — Attention & Growth
*Gary Vaynerchuk standard*

- Mobile is not a consideration — it is the default. Design for the thumb, not the cursor.
- The first 3 seconds decide everything. Hero section: headline, subheadline, CTA above the fold — no exceptions.
- Every section of the landing page must justify its existence by advancing the user toward conversion.
- Social proof is not optional. Testimonials, user counts, trust signals wherever trust needs building.
- Urgency without manipulation. Genuine reasons to act now — never fake countdown timers.
- Every page is a funnel. Know the one action you want and remove every obstacle between the user and that action.

## Part 5 — SEO & Conversion
*Neil Patel standard*

- Every page is a search opportunity. Title tags, meta descriptions, H1s — written for humans first, search engines second.
- Above the fold must include: headline, subheadline, CTA, and one piece of social proof.
- Page speed is a conversion factor. Every second of load time costs users.
- Internal linking creates authority. Pages link to each other with descriptive anchor text.
- Content answers the question the user had when they arrived.
- CTA copy is action-oriented and benefit-led. "Get your free app" outperforms "Sign up."
- Trust signals near every CTA: security badges, testimonials, guarantees, user counts.

**SEO requirements in every generated app:**
- Unique `<title>` per page: "[Page Name] — [App Name]"
- `<meta name="description">` under 160 characters
- One H1 per page, logical H2/H3 hierarchy
- Open Graph tags: `og:title`, `og:description`, `og:image`, `og:url`
- Semantic HTML: `<nav>`, `<main>`, `<section>`, `<article>`, `<header>`, `<footer>`
- JSON-LD structured data appropriate to app type
- `robots.txt` and `sitemap.xml` scaffolded
- Canonical URL tag
- Images lazy-loaded below fold with explicit `width` and `height`

## Part 6 — Usability (Affordance)
*Don Norman — The Design of Everyday Things*

**Affordances** — every interactive element looks interactive. Buttons look like buttons. Links look like links. Inputs look like inputs.

**Signifiers** — labels, icons, and placeholder text signal exactly what to do and where. Every form field has a label above it — never placeholder text alone.

**Feedback** — every action produces visible feedback within 100ms. Five button states designed: default → hover → active → loading → success/error.

**Constraints** — prevent wrong actions. Disable submit until required fields are filled. Date pickers prevent impossible dates. Irreversible actions require confirmation.

**Mapping** — controls match outcomes. Form field order matches the natural order of the task. Navigation order matches the natural flow of the user's journey.

**Conceptual models** — the product behaves the way users expect. Folders contain files. Dashboards show summaries. Never violate these conventions without onboarding to explain why.

## Part 7 — Simplicity
*Steve Krug — Don't Make Me Think*

**The first law:** every page passes the trunk test. A user dropped onto any page instantly knows:
1. What site they're on
2. What page they're on
3. What the major sections are
4. What their options are at this level
5. Where they are in the hierarchy
6. How to search

**Eliminate question marks.** Every label, heading, instruction, and button is unambiguous. If it could mean something else to a first-time visitor — rewrite it.

**Omit needless words.** Remove half the words on every page. Then remove half again. No happy talk. No preamble. No instructions nobody reads.

**Navigation rules:**
- The logo always links to the home page — without exception
- The current page is always highlighted in navigation — without exception
- Back button always works as expected — never break browser navigation
- Navigation is consistent across every page

**Responsive navigation (hard requirement):**
- Desktop (md+): inline links
- Mobile (<md): hamburger icon (Menu) + full-screen drawer
- Drawer: Paper (#f2efe8) background, Ink (#0e0d0b) links, Flame (#FF1F6E) active state, closes on tap/click outside
- No page ships without a working mobile nav
- Verify at 375px before every commit touching nav

**Happy path obsession.** The most common user action on each screen is the most prominent, easiest, fastest to complete. Edge cases are handled gracefully but never given equal visual weight.

## Visila Typography System
Source: Apple HIG Typography PDF (December 2025)
Applied to: visila.com (src/styles/tokens.css) and all generated apps (api/_designSystem.ts)

### Rules (hard requirements)
- Minimum body text: 17px (--text-body). Never below 12px anywhere.
- Font weights: Regular (400), Medium (500), Semibold (600), Bold (700) only
- Never use Thin, Light, or Ultralight — illegible at small sizes
- One typeface per app — hierarchy through weight and size only
- Never use color alone to convey hierarchy
- Letter spacing: negative for body (-0.025em), positive for display (0.012em)
- Line height: 1.5 relaxed for body, 1.2 tight for display

### Token reference (src/styles/tokens.css)
Scale: --text-large-title (34px) through --text-caption (12px)
Weights: --weight-regular through --weight-bold
Tracking: --tracking-display, --tracking-body, --tracking-caption
Leading: --leading-tight (1.2) through --leading-loose (1.6)
Fonts: --font-serif (Playfair Display), --font-mono (DM Mono), --font-sans (Geist Sans)

### Generated apps
Same Apple HIG scale. Font family varies by category (see CATEGORY_TYPOGRAPHY in api/_designSystem.ts). All weights, tracking, and leading values are identical to visila.com tokens.

## Visila Writing Standard
Source: Apple HIG Writing (December 2025) + Visila brand voice
Applies to: all UI copy, Brain hints, error messages, empty states, button labels, placeholders, notifications

### Voice
Co-founder energy. Direct, warm, specific. Honest without being clinical. Never robotic, never condescending, never vague. Think: the smartest person on your team telling you something important in plain language.

### Tone by context
- Building / generating: active, forward ("Building ChefNear...")
- Success: warm, brief ("ChefNear is live.")
- Error: calm, specific, actionable — no blame
- Brain hints: YC partner — specific to THIS edit, never generic
- Empty states: welcoming, directive — always include a next step
- Warnings: direct, no drama, clear what to do

### Capitalization — pick one per element, never mix
- Nav links: ALL CAPS ("HOW IT WORKS", "DASHBOARD")
- Page headings: Sentence case ("Manage your apps.")
- Section headings: Sentence case ("Version history")
- Button labels: Sentence case ("Send my dashboard link →")
- Card labels / badges: ALL CAPS ("ERROR", "COMPLETE", "BUILDING")
- Tooltips / hints: Sentence case
- Never use Title Case anywhere — too formal for Visila's voice

### Button labels — always a verb, always specific
- "Try again" not "Retry"
- "Build my app" not "Submit"
- "Restore this version" not "Revert"
- "Delete app" not "Delete"
- "Update →" is fine — action + direction is clear

### Error messages — no blame, no jargon, clear next step
Pattern: [What happened] · [What to do]
- "Build failed. Check the log or try again."
- "Unable to connect. Check your internet and try again."
- "Enter a valid email address, like name@example.com"
- Never: "Deployment failed" (no next step), "We're having trouble" (vague, uses "we")
- Never: "oops", "uh-oh", "whoops" — insincere
- Never: "we" in error messages — say what happened, not who caused it
- Never show raw API errors to users — translate to plain English

### Possessive pronouns
- Never use "we" in system messages — unclear and evasive
- "your" is fine in direct address ("Your app is live")
- Avoid "my" in button labels — prefer the action alone

### Empty states — always guide, always include a next step
- Dashboard (no builds): "Your first app is one idea away. Describe what you want to build below."
- Version history (no edits): "No edits yet. Every change you make will appear here."
- Brain panel (no hints): "Brain is watching. Hints appear after your first edit."

### Placeholder text — example or description, never robotic
- "name@example.com" not "Enter email address"
- "A marketplace for local chefs..." not "Type here"
- "What do you want to change..." not "Enter your edit"

### Rules checklist — verify before every UI commit
- [ ] Every button label is a verb
- [ ] No "we" in any system or error message
- [ ] No raw API errors shown to users
- [ ] No "oops", "uh-oh", or similar interjections
- [ ] Capitalization consistent per element type
- [ ] Every empty state has a next step
- [ ] Error messages include what happened AND what to do
- [ ] Placeholder text is an example or description, not a command

## Part 8 — Information Architecture
*Rosenfeld, Morville & Arango — Information Architecture: For the Web and Beyond, 4th Edition*

**Organization** — group content and features by user mental models, not technical categories.
- Any core feature reachable in 3 clicks or fewer from the homepage
- Every page belongs to exactly one logical section — no orphan pages
- Navigation labels use the user's words, not the product's internal terminology
- Primary navigation contains 5–7 items maximum

**Navigation systems:**
- User always knows where they are, where they've been, and where they can go
- Breadcrumbs on any page deeper than two levels
- Footer navigation mirrors primary navigation
- Search included on any app with more than 20 pieces of content
- Active states in navigation always visible

**Labeling** — specific and predictable. "Your events" not "Content." "Send update" not "Communicate." Icons always have text labels.

**Search & findability:**
- Filters and facets for any collection of more than 12 items
- Zero results pages suggest alternatives — never a dead end
- Recently viewed items surfaced for logged-in users

**Content structure:**
- Every page has clear hierarchy: primary, secondary, supporting content
- Related content is always linked — users who finish reading one thing have an obvious next thing
- Progressive disclosure for complex features — simple version first, advanced options on demand
- Information scent: every link must clearly signal what the user will find on the other side

## Part 9 — User Story Mapping
*Jeff Patton — User Story Mapping: Discover the Whole Story, Build the Right Product*

**The backbone — user activities** in chronological order:
1. What does the user do first when they arrive?
2. What is the core loop they repeat every time?
3. What does success look like at the end of their journey?
4. What brings them back tomorrow?

The backbone must be visible in the navigation and information architecture.

**Walking skeleton** — the minimum set of tasks that makes each activity work. Build the thinnest possible version that completes the whole journey end to end. Never build a feature-complete first step with an incomplete second step.

**Release slicing** — every generated app ships a complete story:
- User can arrive, understand the product, complete the core loop, and return
- No dead-end pages, no half-built flows, no buttons that go nowhere
- If a feature can't be completed, it doesn't appear in the UI

**User outcomes over features** — every feature maps to a specific user outcome. Features that don't map to an outcome are not generated.

**The story test** — before shipping, walk the journey:
1. User arrives — do they understand what this is within 5 seconds?
2. User signs up — is the flow under 60 seconds?
3. User completes the core action for the first time — is it obvious what to do?
4. User returns the next day — do they know where to pick up?
5. User invites someone or shares — is there a natural moment to do this?

## Part 10 — Product Requirements
*Marty Cagan — Inspired + Sondra Orozco — The Ultimate Guide to PRDs*

**Cagan's four risks — resolved before any feature ships:**
1. Value risk: will users actually want this?
2. Usability risk: can users figure out how to use it?
3. Feasibility risk: can it be built and maintained?
4. Business viability risk: does it serve the business model?

Generated apps default to the simplest possible implementation — complexity is earned through evidence, not assumption.

**Every generated app's README.md contains a living product brief:**

*Problem statement:* Who is the primary user? What problem do they have today? What does success look like for them?

*Outcome goals:* Primary metric (the one number that proves the product is working). Secondary metrics (2–3 supporting signals). Anti-metrics (what we are explicitly not optimizing for).

*Top 5 user stories* in Patton format:
> "As a [specific user], I want to [specific action] so that [specific outcome]."

*Assumptions and risks:* Top 3 assumptions the product is built on. What would invalidate each. Mitigation for the highest-risk assumption.

*Scope:*
- In scope for v1: the minimum feature set that completes the core journey end to end
- Out of scope for v1: everything that can wait — listed explicitly so it's not forgotten
- Never to be built: features that would contradict the core value proposition

*Success criteria:* How will we know in 30 days if this is working?

## Part 11 — Execution & Project Management
*David Allen — Getting Things Done*

**Capture everything.** Every generated app ships with a structured NEXT STEPS section in README.md — the 3 most important actions the owner should take immediately after launch.

**Clarify before acting.** Every feature has a single, unambiguous purpose. Confirmation dialogs clarify consequences. Success states clarify what just happened and what happens next.

**Organize by context.** Dashboards organize tasks by context, not by date: what needs attention now, what is in progress, what is complete.

**Reflect — build in review loops.** Metrics surfaced in the dashboard so the owner can do a weekly review without leaving the product. README includes a weekly review checklist template.

**Engage — next actions always visible.** The dashboard always answers: "What should I do right now?" Every empty state has a single recommended next action. Onboarding is a checklist, not a tour.

**The two-minute rule applied to UX:** any action that takes less than two minutes must be completable inline — never send the user to a separate page for a simple action.

**Waiting for states** explicitly tracked — when something depends on an external action (payment pending, invite accepted), the UI shows it clearly.

## Part 12 — Accessibility
*WCAG 2.1 AA — non-negotiable on every generated app*

- All text meets 4.5:1 contrast ratio (3:1 for large text 18px+)
- All interactive elements have visible focus states — never remove outline without replacing it
- All form inputs have associated `<label>` elements — never placeholder-only
- Color is never the only way to convey information — always pair with text or icon
- All images have alt text; decorative images have `alt=""`
- Keyboard navigation works for all interactive elements in logical tab order
- ARIA roles and labels on custom components (modals, dropdowns, tabs)
- Skip to main content link at top of page
- Font size minimum 16px for body text, never below 12px for any text
- Line height minimum 1.5 for body text
- Touch targets minimum 44×44px with 8px spacing between them

## Part 13 — Security
*OWASP + Defense in Depth — applied to every generated app*

- Principle of Least Privilege on every API route and component
- RLS enabled on every Supabase table with explicit policies
- No secrets or service role keys ever in client code — only `VITE_` prefixed env vars on the client
- Input validation and sanitization on every form field
- Rate limiting on every API endpoint
- Soft deletes on every table (`deleted_at`, `deleted_by`) — never hard delete
- Security headers in every `vercel.json`: `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `Content-Security-Policy`
- `X-Frame-Options`: Set programmatically by the Visila build pipeline. Generated apps must not specify this header — the pipeline sets it correctly for staging (ALLOWALL) to enable iframe preview. Post-claim, users control their own policy.
- No `localStorage` for sensitive data
- `.env.example` committed with all keys documented, no real values

## Part 14 — Nielsen Norman Group
*10 Usability Heuristics — applied to every generated app*

1. **Visibility of system status** — always show loading states, progress, success/error feedback
2. **Match real world** — plain English labels. "Save" not "Persist." "Delete" not "Destroy."
3. **User control** — every flow has a back button or cancel. No dead ends.
4. **Consistency** — same action always looks and behaves the same way
5. **Error prevention** — validate inline before submit, disable submit until required fields are filled
6. **Recognition over recall** — show options, don't make users remember. Use placeholders, helper text, autocomplete.
7. **Flexibility** — simple defaults for new users, power user paths available
8. **Aesthetic and minimalist design** — every UI element must earn its place
9. **Help users recover from errors** — error messages say what went wrong AND how to fix it
10. **Help and documentation** — empty states explain what to do next, never just show a blank screen

## Part 15 — Resilience & Recovery
*iOS Human Interface Guidelines + Material Design — non-negotiable on every generated app*

- Every async operation must have three designed states: loading, success, and error
- No silent failures — every error must be visible, human-readable, and actionable
- Every error message follows the pattern: what happened + what to do next
- No flow ever requires the user to start over — state is always preserved
- Auth and OAuth flows are always resumable — store state before redirecting, restore it on return
- Failed operations always offer a one-tap retry
- Network errors degrade gracefully — show cached state where possible
- Forms never lose user input on error
- Loading states appear within 100ms of any user action
- Never show a blank screen — always show something meaningful

Every generated app is classified before generation begins:

**SIMPLE** — personal site, portfolio, landing page, blog, restaurant site, studio
→ Apply Tiers 1–6, 12, 13, 14, 15

## The Quality Bar

Before finalizing any generated app, every item must pass:

**Design**
- [ ] Would a non-developer be proud to show this to a client?
- [ ] Does every interactive element look interactive without explanation?
- [ ] Are all empty, loading, and error states designed?
- [ ] Does it look and feel like a $10,000 agency built it?

**Copy**
- [ ] Does the landing page pass the 5-second test?
- [ ] Is every piece of copy at 8th grade reading level or below?
- [ ] Has every unnecessary word been removed from every screen?

**Usability**
- [ ] Is every screen's primary action immediately obvious?
- [ ] Can a user dropped onto any page instantly know where they are and what to do?
- [ ] Are all form fields labeled above the input — never placeholder-only?
- [ ] Would a first-time user with no context complete the core task without help?
- [ ] Does every action produce visible feedback within 100ms?

**Information Architecture**
- [ ] Does the navigation reflect how users think, not how the database is structured?
- [ ] Can a new user complete the core loop in under 3 minutes without instructions?
- [ ] Is every core feature reachable in 3 clicks or fewer?

**User Story**
- [ ] Does every feature in the app map to a specific moment in the user's story?
- [ ] Is the primary user's journey complete end-to-end with no dead ends?
- [ ] Can a new user complete the core loop in under 3 minutes without instructions?

**Product**
- [ ] Does the README contain a clear problem statement, outcome goals, and top 5 user stories?
- [ ] Does every feature map to at least one user story?
- [ ] Is there a feedback mechanism so the owner can learn from real users?
- [ ] Does the app instrument the primary metric?

**Execution**
- [ ] Does the README include a NEXT STEPS section with the 3 most important post-launch actions?
- [ ] Does the dashboard answer "what should I do right now?" without explanation?
- [ ] Are all async and waiting states explicitly tracked and visible?
- [ ] Does every empty state have a single recommended next action?

**Accessibility**
- [ ] Does all text meet 4.5:1 contrast ratio?
- [ ] Does every interactive element have a visible focus state?
- [ ] Is color never the only way information is conveyed?

**SEO**
- [ ] Does every page have a unique title tag and meta description?
- [ ] Is there one H1 per page with a logical heading hierarchy?
